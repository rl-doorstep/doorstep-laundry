1. Key Constraints Discovered in the Codebase

These constraints directly shape which tests can run in CI and which require special handling.

ConstraintWhere it livesImpact on tests
Signup requires RESEND_API_KEY src/app/api/auth/signup/route.ts — returns 503 if key is missing Do not test the full signup flow in CI by default. Use seed-created users. For signup-flow tests, mock Resend at the module level or provide a real (cheap) Resend test key.
Auth is credential-only in CI src/lib/auth.ts — Google provider is conditional on env vars Only test credential login. Seed users all have emailVerifiedAt set and password mypass1.
Email verification flow src/app/api/auth/verify-email/route.ts — token stored in DB Can be tested by inserting a token directly via the seed or a setup API call, then calling the verify endpoint with that token.
Address service area src/lib/service-area.ts — returns ok: true when max_service_distance_miles = 0 Leave the setting at 0 (seed default) and omit GOOGLE_MAPS_API_KEY from CI to bypass distance checks in booking tests.
Stripe webhook verification src/app/api/webhooks/stripe/route.ts — requires valid HMAC signature Skip full webhook tests in Phase 1–2. In Phase 2, assert that "Pay" button links to Stripe URL pattern; in Phase 3 add a synthetic webhook test using stripe.webhooks.generateTestHeaderString() at the API layer.
Test seed users are domain-specific scripts/seed.ts — emails use doorsteplaundrylc.com Add a parallel set of playwright+…@doorsteplaundrylc.com users to the test seed, or reuse the existing ones — the seed is idempotent (upsert).
Test DB support is already built scripts/seed.ts reads DATABASE_URL_TEST with --mode test The CI job just needs to set DATABASE_URL_TEST pointing at the ephemeral Postgres service, then run npm run db:seed:test.
Order sequence seeding Test orders use sequence numbers 9001–9008 to avoid collisions E2E tests should use these fixed order numbers to assert against known state.

import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  testDir: './e2e',
  // Reasonable CI timeout; local dev can be shorter
  timeout: 30_000,
  expect: { timeout: 8_000 },

  // Run tests in parallel — each worker gets its own browser context
  fullyParallel: true,
  workers: process.env.CI ? 2 : undefined,

  // Retry once in CI to absorb flakiness from Next.js server cold starts
  retries: process.env.CI ? 1 : 0,

  reporter: process.env.CI
    ? ['list', ['html', { open: 'never' }], ['junit', { outputFile: 'playwright-results.xml' }]]
    : 'list',

  use: {
    baseURL: process.env.PLAYWRIGHT_BASE_URL ?? 'http://localhost:3000',
    trace: 'on-first-retry',
    screenshot: 'only-on-failure',
    video: 'retain-on-failure',
  },

  projects: [
    // Setup project runs once to seed the DB before any tests
    { name: 'setup', testMatch: /.*\.setup\.ts/ },

    {
      name: 'chromium',
      use: { ...devices['Desktop Chrome'] },
      dependencies: ['setup'],
    },
  ],

  // Start the Next.js dev server in CI only when not already running
  webServer: {
    command: 'npm run start',
    url: 'http://localhost:3000',
    reuseExistingServer: !process.env.CI,
    timeout: 120_000,
    env: {
      DATABASE_URL: process.env.DATABASE_URL_TEST ?? process.env.DATABASE_URL ?? '',
    },
  },
});
Why npm run start not npm run dev The start command runs the production build. In CI, build the app first with npm run build, then Playwright launches next start. This is ~3× faster for cold starts than next dev and matches production behavior. Add a build step before the Playwright job.

5. Step 3 — Test Database & Environment Strategy

Local development

Create a second Supabase project (free tier) or a local Postgres database and set DATABASE_URL_TEST in .env. The seed script already routes to it when --mode test is passed.

CI environment

The GitHub Actions job spins up a services.postgres container. This is completely separate from the main lint-and-test job, which has no database at all.

Env var in CIValuePurpose
DATABASE_URLPostgres service connection stringPrisma migrate uses this
DATABASE_URL_TESTSame value as DATABASE_URL in the E2E jobSeed script targets this
NEXTAUTH_SECRETAny 32-char string (GitHub secret)JWT signing
NEXTAUTH_URLhttp://localhost:3000Redirect base URL
STRIPE_SECRET_KEYReal Stripe test key (GitHub secret)Checkout session creation
NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEYMatching test publishable keyClient-side Stripe.js
STRIPE_WEBHOOK_SECRETAny string or skipNot needed until Phase 3 webhook tests
RESEND_API_KEYOmit — see mocking sectionTests bypass signup; notify calls are no-ops
TWILIO_ACCOUNT_SIDOmitSMS silently skipped when key absent
GOOGLE_MAPS_API_KEYOmitService area check returns ok=true when missing+max=0
NEXT_PUBLIC_GOOGLE_MAPS_API_KEYOmitAddress autocomplete degrades gracefully
Why omitting Resend/Twilio/Google Maps is safe
  • sendOrderNotification in src/lib/notify.ts logs a warning and returns { email: false } when RESEND_API_KEY is absent — it does not throw.
  • The Twilio path is guarded by canSendSms which requires both SID and token to be set.
  • Address service area is bypassed when max_service_distance_miles = 0 (the seeded default).
  • The signup route does throw 503 when RESEND_API_KEY is absent — but signup tests use pre-seeded users, not the signup form.

6. Step 4 — Fixtures & Auth Helpers

e2e/fixtures/auth.setup.ts — runs once before all tests

import { test as setup, expect } from '@playwright/test';
import path from 'path';
import { execSync } from 'child_process';

// Run migrations and seed the test DB once per run.
// execSync is acceptable here because setup is single-process.
setup.beforeAll(async () => {
  execSync('npx prisma migrate deploy', { stdio: 'inherit',
    env: { ...process.env, DATABASE_URL: process.env.DATABASE_URL_TEST } });
  execSync('npm run db:seed:test', { stdio: 'inherit' });
});

// Save auth state for each role so tests can start pre-authenticated.
const USERS = [
  { role: 'customer', email: 'ricky+1@doorsteplaundrylc.com',  file: '.auth/customer.json' },
  { role: 'staff',    email: 'ricky+staff1@doorsteplaundrylc.com', file: '.auth/staff.json'    },
  { role: 'admin',    email: 'ricky@doorsteplaundrylc.com',         file: '.auth/admin.json'    },
];
const PASSWORD = 'mypass1.';

for (const u of USERS) {
  setup(`authenticate as ${u.role}`, async ({ page }) => {
    await page.goto('/login');
    await page.fill('#email', u.email);
    await page.fill('#password', PASSWORD);
    await page.click('button[type=submit]');
    // Wait for successful redirect away from /login
    await page.waitForURL((url) => !url.pathname.startsWith('/login'));
    await page.context().storageState({ path: path.join('e2e', u.file) });
  });
}

e2e/fixtures/db.ts — direct Prisma access for test state manipulation

/**
 * Direct Prisma client for test setup helpers.
 * Uses DATABASE_URL_TEST so it targets the isolated test database.
 * Import from test files to create or modify data without going through the UI.
 */
import { PrismaClient } from '@prisma/client';

const prisma = new PrismaClient({
  datasources: { db: { url: process.env.DATABASE_URL_TEST } },
});

export { prisma };

/** Get a seeded order by its fixed sequence suffix (9001–9008) */
export async function getTestOrder(seqNum: number) {
  // Test orders use orderNum(seqNum) = "ORDER-YYYYMMDD-seqNum"
  return prisma.order.findFirst({
    where: { orderNumber: { endsWith: `-${String(seqNum).padStart(4, '0')}` } },
    include: { orderLoads: true, pickupAddress: true, deliveryAddress: true },
  });
}

export async function getPastDueOrder(customerEmail: string) {
  const user = await prisma.user.findUnique({ where: { email: customerEmail } });
  if (!user) throw new Error(`User not found: ${customerEmail}`);
  // Create a synthetic past-due order for blocking tests
  return prisma.order.create({
    data: {
      orderNumber: 'TEST-PASTDUE-001',
      customerId: user.id,
      /* ... minimal required fields ... */
    },
  });
}

8. Step 6 — Phase 2 Tests: Booking & Dashboard

Strategy for booking form The booking form uses an interactive date picker (buttons with day labels) and address verification. In CI without Google Maps, address autocomplete degrades to plain inputs. Use a known in-service-area address (Las Cruces, NM) and omit the Google Maps key so the service area check is skipped entirely.

e2e/booking/booking-flow.spec.ts — annotated skeleton

test('complete 3-step booking flow', async ({ customerPage: page }) => {
  await page.goto('/book');

  // Step 1 — default is 1 load panel; verify it exists
  await expect(page.getByText('Load 1')).toBeVisible();

  // Add a second load
  await page.getByRole('button', { name: /Add load/i }).click();
  await expect(page.getByText('Load 2')).toBeVisible();

  // Max load guard — click 8 more times to reach 10
  for (let i = 0; i < 8; i++) {
    await page.getByRole('button', { name: /Add load/i }).click();
  }
  await expect(page.getByRole('button', { name: /Add load/i })).toBeDisabled();

  // Pick the first available pickup date (enabled button in the date picker)
  await page.locator('button:not(:disabled)', { hasText: /Mon|Tue|Wed|Thu|Fri|Sat|Sun/ })
    .first().click();

  await page.getByRole('button', { name: /Continue/i }).click();

  // Step 2 — address form (no Google Maps in CI)
  await page.getByPlaceholder(/Label/i).fill('Home');
  await page.getByPlaceholder(/Street/i).fill('2001 E Lohman Ave');
  await page.getByPlaceholder(/City/i).fill('Las Cruces');
  await page.getByPlaceholder(/State/i).fill('NM');
  await page.getByPlaceholder(/ZIP/i).fill('88001');

  await page.getByRole('button', { name: /Create order/i }).click();

  // Should redirect to /orders/[id] on success
  await page.waitForURL(/\/orders\//);
  await expect(page.getByText(/scheduled/i)).toBeVisible();
});

Key booking sub-tests to cover

  • Booking with 0 loads is impossible (the "Add load" button always starts with 1 and remove stops at 1)
  • Max 10 loads — "Add load" button becomes disabled
  • Delivery date cannot be before pickup date — enforced by earliestDeliveryDate in the component; buttons are disabled
  • Past-due block — seed a delivered+unpaid order for customer 2, then attempt booking as customer 2 and assert the amber block message appears

10. Step 8 — GitHub Actions CI Job

Add this job to .github/workflows/ci.yml alongside the existing lint-and-test job. It only runs on PRs targeting dev. The existing job is unchanged.

# .github/workflows/ci.yml — full file (existing job + new E2E job)
name: CI

on:
  push:
    branches: [main, dev]
  pull_request:
    branches: [main]

jobs:
  # ── Existing job (unchanged) ────────────────────────────────────────
  lint-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"
      - name: Install dependencies
        run: npm ci
      - name: Lint
        run: npm run lint
      - name: Generate Prisma client
        run: npx prisma generate
      - name: Test
        run: npm run test

  # ── New E2E job ─────────────────────────────────────────────────────
  e2e:
    # Only run on PRs to dev, not on pushes to main (protect production gate)
    if: github.event_name == 'pull_request' && github.base_ref == 'dev'
    runs-on: ubuntu-latest
    # Non-blocking during Phase 1 ramp-up — flip to false once stable
    continue-on-error: true

    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_USER: pw_test
          POSTGRES_PASSWORD: pw_test
          POSTGRES_DB: laundry_test
        ports:
          - 5432:5432
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5

    env:
      # Both vars point at the same CI Postgres (the seed script reads DATABASE_URL_TEST)
      DATABASE_URL: postgresql://pw_test:pw_test@localhost:5432/laundry_test
      DATABASE_URL_TEST: postgresql://pw_test:pw_test@localhost:5432/laundry_test
      NEXTAUTH_SECRET: ${{ secrets.NEXTAUTH_SECRET }}
      NEXTAUTH_URL: http://localhost:3000
      STRIPE_SECRET_KEY: ${{ secrets.STRIPE_SECRET_KEY_TEST }}
      NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY: ${{ secrets.STRIPE_PK_TEST }}
      # Intentionally omitted: RESEND_API_KEY, TWILIO_*, GOOGLE_MAPS_API_KEY
      # These services degrade gracefully when keys are absent (see code analysis)
      PLAYWRIGHT_BASE_URL: http://localhost:3000
      CI: true

    steps:
      - uses: actions/checkout@v4

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      - name: Install dependencies
        run: npm ci

      - name: Install Playwright browsers (Chromium only)
        run: npx playwright install --with-deps chromium

      - name: Generate Prisma client
        run: npx prisma generate

      - name: Run database migrations
        run: npx prisma migrate deploy

      - name: Build Next.js app
        run: npm run build

      - name: Run Playwright E2E tests
        run: npx playwright test

      - name: Upload Playwright report
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: playwright-report
          path: playwright-report/
          retention-days: 7

      - name: Upload JUnit results
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: playwright-junit
          path: playwright-results.xml

Build time in CI Running npm run build before Playwright adds ~2–3 minutes to the job. The tradeoff is that you test against production-optimized code (correct behavior) rather than dev-mode code (hot reload artifacts can affect timing). An alternative is to run next dev instead of next start — faster startup, but slightly different behavior. Recommended: use next start from the start; it keeps CI results credible.
